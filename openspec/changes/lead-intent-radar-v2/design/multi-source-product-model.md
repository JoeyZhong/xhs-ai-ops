# 多信源产品/数据模型（lead-intent-radar V2）

> C1 交付：把猪八戒/知乎接入后，**它们的信息如何与 goal / persona / lead 等现有实体关联**讲清楚，作为后续 UI 设计（用户自做）与实现的产品依据。

## 0. 一句话

**信源是"入口"，不是新实体。** 猪八戒、知乎和小红书一样，只是同一个 goal 下的**多个采集入口**；三者产出的线索进**同一个收件箱**，用 `lead.source` 区分。判定逻辑、客户画像、人设全部**共享**；**只有两处按信源不同**：草稿的"文体"、触达的"动作"。

## 1. 实体关系（现有实体不新增，只加一个 goal 配置）

```
tenant(default)
  └── goal「获取快速审计出表客户」  ← 业务目标（不变）
        ├── keywords          监控词（不变，三源共用）
        ├── persona_id        审计人设（不变，三源共用口吻）
        ├── target_audience   客户画像（不变，判定依据）
        └── lead_sources ★新增  = ["xhs","zhihu","zhubajie"]   ← 唯一的新配置
                 │
                 ├─ xhs 采集 ─┐
                 ├─ zhihu 采集─┼→ Signal[](归一) → intent.classify(同一判定)
                 └─ zhubajie ─┘                          │
                                                   qualified?
                                                         │
                                            lead(source=xhs/zhihu/zhubajie)
                                                         │
                                              同一个 /admin/leads 收件箱
```

**关键点：goal 与信源是 1:N。** 一个 goal 配 `lead_sources` 决定它从哪些池子捞线索。换垂直行业时（如法律咨询），还是这套：换 goal 的 keywords/画像/persona + lead_sources，引擎不动。

## 2. 三个平台的原始信息 → 统一 Signal 的字段映射

各平台原生形态不同，但**归一到同一个 Signal**，平台特有字段塞进 `meta`，不污染主流程：

| Signal 字段 | 小红书(帖) | 知乎(问题) | 猪八戒(需求单) |
|---|---|---|---|
| `source` | `xhs` | `zhihu` | `zhubajie` |
| `source_url` | 笔记链接 | 问题链接 | 需求单链接 |
| `author` | 发帖人昵称 | 提问者 | 雇主/发单人 |
| `post_text` | 笔记正文 | 问题标题+描述 | 需求标题+需求描述 |
| `posted_at` | 发布时间 | 提问时间 | 发单时间 |
| `signal_key` | `xhs:<hash>` | `zhihu:<hash>` | `zhubajie:<hash>`（前缀含源，跨源天然唯一） |
| `meta`（平台特有，可选） | — | 关注数/回答数 | **预算 / 交付周期 / 是否已接单**（猪八戒结构化字段） |

> 设计含义：UI 主信息（队列卡、原帖区）用通用字段即可三源统一渲染；猪八戒的"预算/周期"等可作为 `meta` 徽章**选择性展示**（你设计时定要不要露）。

## 3. 不变的部分（三源共享，无需分支）

| 概念 | 是否随信源变 | 说明 |
|---|---|---|
| `intent.classify` 判定 | **不变** | 判的是"这段文本是不是求审计的真实客户"，与平台无关。猪八戒需求单通常 is_intent 几乎必真，但画像匹配度仍照打 |
| `trigger_type`（贷款/投标/高新/外资/注销） | **不变** | 描述的是**买家处境**（为什么要审计），跟他在哪个平台发无关 |
| `match_score` / 画像 | **不变** | 从 goal.target_audience 派生，三源同一把尺 |
| `persona`（审计人设） | **不变** | 三源同一个口吻人设 |
| 维度隔离（tenant/goal_id） | **不变** | lead 仍按 goal_id 进对应收件箱 |
| lead 生命周期 | **不变** | detected→qualified→drafted→pending→touched→skipped |
| 引流词/雷同度校验 | **不变** | 三源草稿都过同一套红线校验 |

## 4. 按信源不同的部分（只有两处）

### 4.1 草稿"文体"随源变（persona 不变，prompt 模板变）

同一个审计人设，但回不同平台该用不同文体：

| 源 | 草稿文体 | 长度/风格 |
|---|---|---|
| 小红书 | 评论区**短回复** | 80–130 字，像同行随手点评（V1 已实现） |
| 知乎 | **专业回答** | 较长、有条理、先给干货建立信任后留口子 |
| 猪八戒 | **接单报价话术** | 直给：能做什么、大概多久、价位档，贴合"接单"语境 |

→ 实现上：`outreach.draft` 按 `source` 选 prompt 模板；人设 system_prompt 不变。

### 4.2 触达"动作"随源变 —— **本轮只有小红书做自动发**

各平台原生触达动作根本不同，本轮**只接小红书的自动发**，知乎/猪八戒仍走 V1 的"复制草稿+打开原帖"人工动作：

| 源 | 原生触达动作 | 本轮 V2 怎么做 |
|---|---|---|
| 小红书 | 在帖下发评论 | **一键发送**（ReaJason `comment_note`，dryrun 默认）✅ 写端本轮做 |
| 知乎 | 写回答 / 评论 | **复制草稿 + 打开原帖**（人工，沿用 V1）——写端本轮不做 |
| 猪八戒 | 接单/提交报价（结构化，非评论） | **复制草稿 + 打开需求单**（人工，沿用 V1）——接单是结构化动作，不是"发评论"，本轮不自动化 |

> 设计含义：收件箱详情区的**主按钮按 `lead.source` 切换**——xhs 显示「一键发送」，zhihu/zhubajie 显示「复制草稿+打开原帖」。这是 UI 上最需要你在设计稿里体现的分支。

## 5. 给你的设计输入清单（你在桌面端据此设计 UI，我不动）

收件箱在 V1 基础上，V2 需要你的设计覆盖这几点变化：
1. **队列卡**加 `source` 标识（小红书/知乎/猪八戒，图标或文字标签）+ 可选源筛选 tab。
2. **猪八戒线索**的 `meta`（预算/交付周期）要不要在卡片或详情露出。
3. **详情区主按钮按源切换**：xhs=一键发送（+ dryrun/已发 状态）、zhihu/zhubajie=复制草稿+打开原帖。
4. **一键发送的状态反馈**：发送中 / 已发(显示平台返回) / 校验未过禁发 / 引擎未就绪(dryrun 提示) / 速率超限(今日已达上限)。
5. 顶部 stats 是否按源拆分（可选）。

> 我**不**产出 mockup HTML、**不**碰 design-sync。以上是文字版产品约束，供你设计参考。

## 6. 配置层落点（实现时）

- `goal.lead_sources: ["xhs","zhihu","zhubajie"]`（缺省 `["xhs"]`，向后兼容 V1）。
- `lead.source` 已存在（V1 即有），三源复用。
- `lead.meta` 存平台特有字段（猪八戒预算等）。
- `goal.lead_radar_enabled` 不变，控制是否被 scheduler 周期扫。
